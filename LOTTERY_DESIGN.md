## 0. 背景與目標

- **問題**：鑽石目前唯一出口是 P 系列車款（5 台）。對車款外觀沒興趣的玩家，鑽石
  等於沒有消費動機，IAP 鑽石購買的吸引力被限縮在「想換車」的一小群人。
- **目標**：① 新增一個抽獎轉輪玩法，提升每日回訪誘因；② 順帶開闢幾個新的鑽石
  消費出口，但**不能賣任何影響排行榜公平性的東西**（不能賣額外排名賽次數、不能
  賣任何提升分數的道具）——只能是外觀/展示/便利性道具，延續本專案一路的反作弊
  原則。
- **不新增 IAP 商品**：現金儲值管道維持現狀（買鑽石 3 包 + 永久去廣告），本次全部
  新道具都用鑽石購買，鑽石獲取管道也不變（排行榜名次獎勵 + IAP 購買）。

## 1. 抽獎機制總覽

- 每日 **2 次免費抽獎**，每次要先看一次獎勵廣告才能抽。
- 免費次數用完後，按鈕變成「**20 鑽石 / 次**」，可以無限次用鑽石抽（不限每日次數）。
- 呈現方式：**拉霸機**（沿用 `RandomSlot.tsx` 的視覺語言，質感與音效另外升級，跟
  選賽道的拉霸機做出區隔），機率不會透過畫面元素的大小暴露出來（跟輪盤不同，
  輪盤格子大小會暗示機率，拉霸機純粹是捲軸停格，觀感上更乾淨）。
- 停格瞬間依獎項稀有度分級加特效：
  - 一般鑽石獎項：數字彈出動畫即可。
  - P 系列車款／黑天鵝：加螢幕震動 + 對應主題光效（P 系列沿用金色系、黑天鵝用
    黑金配色）+ 音效升級。
- 旁邊有一個「機率表」按鈕，點開顯示下表（Google Play 政策要求 loot box 機制要能
  查到機率，見下方「政策合規」章節）。

### 重複保護（抽到已擁有的車款）

抽到已經擁有的 P 系列車款或黑天鵝，自動換算等值鑽石補償，不浪費這次抽獎：
- P 系列：補償 = 該車款自己的鑽石售價（例如重複抽到 P1 補 300 鑽）。
- 黑天鵝：補償 **800 鑽石**（明顯高於任何 P 系列車價，反映稀有度；此數字為提案，
  動工前可再確認）。

## 3. 黑天鵝隱藏車款

- 車庫新增第三個分類（現有：🎯 成就車款、💎 鑽石車款），標題「**🖤 隱藏車款**」。
- **取得前**：卡片顯示車輛剪影（全黑輪廓，看不出配色細節，純程式碼濾鏡處理，
  不用另外做美術），搭配一個神秘符號（提案：❓或鎖頭），不顯示「進度」（跟成就
  車款的達成度進度條不同——黑天鵝是純機率獲得，沒有「進度」這個概念）。
- **取得後**：正常顯示，跟其他已擁有車款一樣可裝備。
- 只能透過抽獎取得，不開放直接用鑽石購買（維持稀有感）。
- 車輛風格：**「黑天鵝」**——台股術語（極端罕見事件），視覺建議走神秘黑金配色，
  跟 P 系列的「黑金巡航」（p3-gold）區隔開，可以更暗、更少金色比例，凸顯「異類」
  質感。風格定案後直接交給 Grok 出圖，走現有 `Grok_Original` → 手動去背 → OpenCV
  量測流程（跟 P/Q 系列一致）。
- `garage.ts` `BIKE_SKINS` 新增一筆：`{ id: "hidden-blackswan", name: "黑天鵝",
  price: 0, locked: true, ... }`（`locked:true` 沿用 Q 系列的「不自動擁有」機制，
  但解鎖來源是抽獎 RPC 不是成就系統）。

## 8. 技術要求（不可妥協）

1. **抽獎結果一律伺服器 RPC 決定**：機率表寫死在 SQL 函式裡，前端只負責送出
   「要不要用鑽石抽」這個布林值 + 播放拿到的結果動畫，不能自己決定/驗證中獎結果。
2. **每日 2 次免費次數伺服器端計數**：比照 `wallet_daily_attempts` 模式，新表
   `wallet_daily_lottery`，不能只靠前端 localStorage 計數（本專案已經在
   `consume_attempt()` 42702 事故 + Capacitor 換殼 localStorage 歸零兩次踩過這個坑）。
3. **鑽石/票券扣款與發獎必須同一個 transaction 內原子化完成**（同一支 RPC 內用
   `for update` 鎖列，避免前端顯示扣了、後端沒扣到的競態）。
4. **PL/pgSQL `returns table(...)` 命名地雷**：新函式的輸出欄位（`diamonds`、
   `tickets`、`owned` 等）容易跟 `player_wallet` 資料表欄位同名，**所有 SELECT/
   UPDATE 一律要幫 `player_wallet` 加表別名**（`w.diamonds` 不能寫裸的
   `diamonds`），這是本專案踩過兩次的 42702 地雷（`wallet_earn`、
   `consume_attempt`），這次設計 SQL 時已經全程加別名，之後如果要再擴充也要
   維持這個習慣。
