# TaiexRider 遊戲模式與長期留存規劃（2026-07-02 定案方向）

> 使用者已看過本方向分析並同意「**全部保留、分批實作、不用一次做**」。
> 每批實作前從本檔勾選項目即可。封測期間求穩：牽動 schema/RPC 的項目一律等使用者點頭才動。

---

## 診斷：為什麼只有每日排名賽留得住人

| 模式 | 留存鉤子 | 缺口 |
|------|----------|------|
| 每日排名賽 | ✅ 內容每日換（真實盤勢）＋競爭每日重置＋5 次稀缺性 | — |
| 自選賽道／長征 | 內容量大（~1000 支） | ❌ 無比較機制、無回訪理由 |
| 隨機拉霸 | 開獎的變動獎勵感 | ❌ 開完獎後與自選同一個問題 |
| 經典模式 | 固定地形＋永久紀錄 | ❌ 單一保持者＝一人接近極限後對所有人「宣告死亡」，與每日賽的「每天重置」正好相反 |

**核心結論**：每日排名賽 = Wordle 結構（全服今天同一題＋成績可炫耀＋明天重來）。策略是**把這條最強的線加粗**，其他模式用「跟自己比／收集」補迴圈，而不是每個模式都硬塞一個排行榜。

**節制原則**：留存機制堆太多會互相稀釋注意力。每批只上 1~2 個，用監控數據（events 表）驗證有效再上下一批。

## 借鑑對象 → 對應設計

- **Wordle**：全服同題＋emoji 分享卡 → 分享文案連動當日真實盤勢（本作獨有優勢：內容跟現實世界掛鉤，天生有話題）。✅ v0.12.5 已上分享按鈕（第一版文字），分享「卡」視覺化是下一步。
- **Duolingo**：streak 連續天數（損失規避）＋週聯賽小分組（30 人一組有感競爭）。
- **Trackmania**：每關官方獎牌（銅/銀/金）——固定關卡不靠榜也有目標，正解經典模式霸榜問題。
- **Dark Souls**：全服死亡血跡 → **死亡熱點**：監控 events 已在收 `death.xr`（位置），同一份數據疊回賽道顯示「今天已有 N 人死在這個跳空缺口」＝監控數據直接變遊戲內容與社群哏，一份工投三用。
- **Hill Climb Racing**：元進度收集 → **股票圖鑑**（騎過 87/1090 支），把自選/長征綁進留存迴圈。
- **手遊週任務**：任務的真正作用是把玩家導流到不會主動去的模式。

## 三種玩家心態全覆蓋

**競爭比較型**：經典模式改 Top N＋百分位回饋（「超越 78% 玩家」）；經典週榜與永久榜並存；個人最佳突破提示（最低成本的競爭感）；好友比較（長期）。
**看戲吃瓜型**：死亡熱點/今日最扯死法側榜；Ghost 鬼影賽跑（資料設計與反作弊第四層共用，見 ANTICHEAT_DESIGN.md）；排行榜 emoji 反應（極簡社交）。
**累積收集型**：streak 連續參賽；股票圖鑑；終身成就（總翻轉/總里程/造訪股票數）；連續登入解鎖車型 sprite（呼應 BETA #2 多車型）。

## 實作批次（成本效益排序）

### 第一批（便宜、零 schema 風險，封測期可上）
| 項目 | 成本 | 說明 |
|------|------|------|
| 個人最佳突破提示 | 極小 | localStorage 記 PB，結算畫面「🎉 打破個人紀錄！」＋分享文案帶入 |
| 經典模式獎牌制 | 小 | 12 關 × 銅/銀/金分數門檻（constants 定義），localStorage 記，選單卡片顯示獎牌，36 個短期目標 |
| streak 連續參賽 | 小 | localStorage 記連續天數（用 resolveSessionDate 對齊連假），每日賽頁顯示 🔥N |
| 分享卡視覺化 | 中 | Canvas 畫成績卡（走勢圖+成績+日期）→ navigator.share files / 下載圖片 |

### 第二批（需少量 schema，等使用者點頭）
| 項目 | 成本 | 說明 |
|------|------|------|
| 全服死亡熱點 | 小～中 | events 已有數據；加一支「當日死亡分佈」彙總 RPC（匿名聚合，無隱私問題）＋賽道渲染骷髏標記 |
| 經典模式 Top N + 百分位 | 中 | classic_records 改多列（schema/RPC 變更），UI 重用每日榜 |
| 經典週榜 | 中 | 同上加 week key 欄位 |
| 週任務 | 中 | 任務定義純前端（seeded 週目標），完成狀態 localStorage；先不做伺服器獎勵 |
| 股票圖鑑 | 中 | run_start events 或 localStorage 記騎過的 code；圖鑑頁 + 完成度 |

### 第三批（長期，工程量大）
| 項目 | 說明 |
|------|------|
| 週聯賽分組 | 30 人小組升降級；等 DAU 有規模才有意義（分組人數不足會很空虛），用監控數據決定時機 |
| Ghost 鬼影賽跑 | 依賴反作弊第四層的操作序列資料設計，兩邊一起做 |
| 排行榜 emoji 反應 | 需 schema + 防濫用，排最後 |
| 好友邀請比較 | 分享連結帶對戰標記，需後端配對 |

### 驗證方式
每批上線兩週後看 events 數據：次日留存（analytics_queries.sql 6️⃣）、各模式開局分佈（2️⃣）是否改善。沒改善的機制要敢下架，避免 UI 堆積。
